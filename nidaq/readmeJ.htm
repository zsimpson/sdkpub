NI-DAQ 6.9.1からNI-DAQ6.9.2への変更点

NI-DAQ 6.9.2のアップデートの主な目的は、新しいデバイスおよびWindows XPへのサポートを追加する点です。このリリースには、NI-DAQ 6.9.1からのいくつかの問題点を修正するNI-PALのアップグレードも含まれます。

NI-PALで修正される問題点

NI-PALでは次の機能を提供します。

  • 2つの同一のPCMCIA DAQCardのインストールおよび構成が、Windows 2000で可能になりました。
  • Windows NTでPCMCIA DAQCard用のCardWizard CardおよびSocket Servicesサポートを追加します。このサポートにより、一度に一つのNational Instruments DAQCardデバイスがインストールされるようになります。
  • これはnipalu.dllを2番目以降のプロセスでロードするのを妨げる、DLL競合エラーを防止します。National Instrumentsは、第3者のDLLと競合を避けるため、nipalu.dllの優先ロードアドレスを変更しました。

NI 4451で転送状態を構成する

NI 4451デバイスでは、AO_Change_Parameter関数を使用して、AO FIFO転送状態を構成できます。それぞれの選択可能な転送状態では、デフォルトのFIFO 転送カウントが使用されます（FIFO 転送カウントはプログラムできません）。AO FIFO転送状態とデフォルトのFIFO転送カウントの情報については、『NI-DAQ Function Reference Help』を参照してください。

複数のDSAデバイスを同期する

DSAデバイスを同期するには、次の点に注意してください。

  • DSAデバイスは、同じタイプのDSAデバイスと一緒に同期することができます。例えば、NI 4772を同期するには、他のNI 4772デバイスを使用しなければなりません。
  • 1394用NI 4472では、スレーブデバイスに、開始トリガ信号を送信するために、RTSI 0、RTSI 1、またはRTSI 2を使用しなければなりません。複数の1394用NI 4472を同期するための残りの手順については、PCI-DSAデバイスの項目を参照してください。詳細については、「NI-DAQ Function Reference Help」の「Synchronizing Multiple PCI-DSA Devices」セクション、またはプログラミング方法についてはLabVIEWヘルプを参照してください。
  • LabVIEWヘルプのSynchronizing Multiple PCI-DSA Devicesトピックの、あるサンプルのダイアグラムにエラーがあります。スレーブデバイスでは、AI Start VIに配線されているトリガソースは、RTSI 0ではなく、RTSI 1であるべきです。このサンプルのダイアグラムでは、マスタデバイスはRTSI 1で、AI Start Trigger VIを制御するので、スレーブデバイスはRTSI 1上で、AI Start Trigger VIを受信しなければなりません。

NI-DAQ 6.7以降のWindows対応LabVIEWデータ集録ライブラリ（LLB)の変更点

NI-DAQ6.7以降では、以下のリストにあるWindows対応LabVIEWのデータ集録ライブラリ（LLB）（<LabVIEW>\vi.lib\daqのフォルダにインストールされる）は、LVDAQ.DLLというダイナミックリンクライブラリ（DLL）から関数を呼び出します（このDLLはLLBと同じディレクトリにあり、以前は各VIに組み込まれていたコードインタフェースノード(CIN)の代わりとなっています）。

  • _daq301.llb
  • _nidaq32.llb
  • dqchnutl.llb
  • misc.llb
  • zadvd.llb
  • zadvdctr.llb

これらLLBとDLLはWindows対応LabVIEW5.x、LabVIEW/RTバージョン5.1.2以降、BridgeVIEW 2.x、および学生版LabVIEWバージョン5.xに対応しています。尚、これらLLBはWindows対応NI-DAQバージョン5.0以降に対応しています（実行ボタンの矢印が壊れずにロードします）。サポートされていない機能がある場合は、VIが実行された際に、DAQエラーコード-10242（functionNotFoundError)を返します。

CINではなくDLLを呼び出すことにより、VIのロード時間が大幅に短縮されます。ベンチマークとしまして、Windows NT 4.0 SP4搭載のPentium III500 MHz Dell Precision 610でテストした結果、NIDAQ32.DLLがメモリにロードされた後に、最初のDAQ VIがLabVIEWにロードされる時間が平均で75%削減されました。

以下のWindows対応LabVIEWの関連ファイルは、NI-DAQ 6.7以降のデータ集録LLBを使用している場合は必要ありません：

  • <LabVIEW>\vi..lib\flatten.lsb (LVDAQ.DLLに含まれています)
  • <LabVIEW>\daqdrv (同じ機能がLVDAQ.DLLに含まれています)

メモ：      LabVIEWアプリケーションビルダでDAQライブラリVIを呼び出しているEXEファイルを作成している場合、ビルドスクリプトにLVDAQ.DLLの参照を手動で追加することが必要な場合があります。

Microsoft Visual C++でのNI-DAQサンプルプログラムのコンパイルおよび実行に関する情報

サンプルをコンパイルし、実行するには、以下の手順に従ってください：

  1. File→Newから、新しいプロジェクトを開きます。Projectsタブが、デフォルトで出てくるはずです。
  2. Win32 Console Applicationをプロジェクトタイプとして選択し、プロジェクト名を記入します。
  3. Empty Applicationを選択します。新規プロジェクトを作成したら、フォルダメニューを左に、ファイルスクリーンを右に表示します。プロジェクトにあるファイルをダブルクリックすると、右側にそのファイルが表示されます。ナビゲーションを簡単にするには、Class ViewからFile Viewに変更します。
  4. Project→Add to Project→Fileを選択します。
  5. NI-DAQ\LIBzディレクトリを選択し、nidaq32.libファイルとnidex32.libファイルを追加します。またNI-DAQ\Includeフォルダから、nidaqex.hとnidaq.hファイルを追加します。2つのライブラリおよびヘッダファイルをプロジェクトに追加すると、ほとんどのデータ集録アプリケーションのサンプルに、ソースコードを統合します。
  6. Tools→OptionsからDirectoriesタブを選択します。
    1. Include Filesにオプションを設定します。NI-DAQ\Includeに同じディレクトリを指定します。
    2. 同じタブから、Librariesにオプションを設定して、NI-DAQ\LIBにパスを設定します。
  7. Easy I/O関数を使用するアプリケーションを使用する場合、Easy I/O関数は、CVI Run-Timeライブラリからいくつかの関数を使用するので、追加ファイルを含む必要があります。追加ライブラリは、easyio.lib、cvisupp.lib、cvirt.lib、そしてcvimain.libです。これらはCVI\EXTLIBディレクトリに存在します。
    1. Tools→OptionsからDirectoriesタブを選択します。
    2. Include Filesにオプションを設定します。CVI\INCLUDEディレクトリを指定します。
    3. 同じタブから、Librariesにオプションを設定して、CVI\EXTLIBにパスを設定します。
  1. 使用するプロジェクトに必要な場合、ソースコードとヘッダを追加します。

これでプロジェクトを実行ファイルに作成することができます。

デバイスがMeasurement & Automation Explorer に表示されない場合の注意点

Measurement & Automation Explorer を使用していて、使用中のデバイスが表示されない場合、F5ボタンを押すか、表示→最新の情報に更新を選択して、Measurement & Automation Explorer をリフレッシュします。

Windows XP をご使用でデバイスが表示されない場合、コントロールパネル→システムを開いてハードウェアタブを選択します（またはコントロールパネル→管理ツール→コンピュータの管理を開きます）。 デバイスマネージャ ボタンを選択してその他のデバイスを選択したら、表示されているデバイスを右クリックしてドライバ タブを選択します。ドライバの更新 …を選択します。ドライバがロードされたら MAX を開いてF5 を押すか、表示→最新の情報に更新を選択し、MAX をリフレッシュします。

DAQPad-6020Eの注意事項

外部タイミングによるA/D:DAQPad-6020Eは、外部サンプルインターバルおよびスキャンインターバルタイミングを低速(すなわち最大約90 Hz)でサポートします。より高速で外部からサンプリングするには、情報タイプ： 割り込み生成モード、情報設定：FIFO半分毎に割り込みを生成に設定します。

ソフトウェアアナログトリガ：DAQPad-6020Eは、低速集録（最大約75サンプル/秒）の場合のみ、ソフトウェアアナログトリガをサポートします。この制限は他のタイプのトリガには適用されません。アナログトリガについては、「NI-DAQ Function Reference Manual」の「Config_DAQ_Event_Message」の項目、またはLabVIEWのデータ集録ヘルプのAI Trigger Config.viの項目を参照してください。

連続低速AI転送：DAQPad-6020Eを使用すると、連続低速AI転送(すなわち、80から約400サンプル/秒)の開始時に、データが集録バッファに入ってくる前に顕著な遅延が見られます。この理由は、A/D変換が生成されても、AI FIFOが少なくとも半分満たされるまでデバイスはPCへのデータ転送を開始しないためです。

タイミングによる非バッファ型AI：DAQPad-6020Eでは、LabVIEWのサンプル、Cont Acq&Chart (hw timed).vi で説明するようなタイミングによる非バッファ型AI操作は、約50 Hzに制限されます。それより高速の場合、NI-DAQがデバイスから一度に1スキャンずつ読み取ってAI FIFOを空にしようとする際、LabVIEWが応答しなくなることがあります。

DAQPad-6070E/DAQPad-6052Eの注意事項

1394アダプタボードの中には、不正確なFIFOサイズを報告するものがあります。したがって、FIFOサイズには最小公分母を使用しています。1394アダプタボードのFIFO深度が456バイト以上であったり、FIFO が誤った速度を報告する場合は、弊社のftpサイトのftp://ftp.natinst.com/support/daq/pc/ieee1394からfwctl.exeおよびreadme.txtをダウンロードして、性能を向上させる必要があります。

通常、制御アプリケーション内のボードとmini-MITE FIFOを無効にして、速度を改善することができます。ただし、この方法はDAQPad-6070E上のすべてのDMAチャネル転送に影響を及ぼします。また、FIFOサイズはDMA転送を実行する前のみ変更が可能です。

NI 6527に関する情報

6527デバイスの割り込みレベル（IRQL）が、USBホストコントローラまたは1394ホストコントローラのIRQLと一致する場合、検出の変更はできない可能性があります。この問題とそれを回避する方法については、「PCIデバイスとWindows 98で割り込みを使用する際の問題点」を参照してください。

NI 6534に関する情報

サポートされる新機能

改善バッファ出力操作：6534デバイスには、メモリモジュールがボードに2つ、1グループにつき1つあります。デフォルトでは、出力操作を開始する前に、NI DAQが全データバッファか全メモリモジュールのどちらか小さい方でSCARABをプリロードして、バスの帯域幅制限を除去します。ただし、デフォルトのプリロードに上書きすることもできます：

  • LabVIEWではDIO Parameter VIをParameter name = SCARABs Preload Enable、Value in = Offで実行します。
  • NI-DAQ APIで、C言語でプログラムしている場合、set_DAQ_Device_InfoをinfoType = ND_FIFO_TRANSFER_COUNT、infoValue = ND_NONEで呼び出します。

リピートデータの連続高速出力：6534デバイスは、出力データがSCARABに一度だけロードされる出力メモリのループモードをサポートします。バッファサイズがオンボードメモリと同じサイズかそれより小さい場合、全バッファは何度も繰り返し使用されます。

  • LabVIEWでは、Parameter NameをPattern Generation Loop Enableに設定して、DIO Parameter VIを実行します。そして、再生成モードをイネーブルにし、連続出力向けの一般的なDIO構成に設定します。
  • C言語でプログラムしている場合、infoType = ND_PATTERN_GENERATION_LOOP_NABLE、infoValue = ND_ONでset_DAQ_Device_Infoを呼び出します。

オンボードメモリサイズ：6534デバイスのメモリモジュールには、さまざまなサイズがあります。6534デバイスのメモリサイズを調べるには、下記の方法があります：

  • LabVIEWでは、Information type = SCARAB A Memory Size (group 1)またはScarab B Memory Size (group 2)として、Get DAQ Device Information VIを実行します。
  • 他の開発環境を使用している場合、info Type = ND_SCARAB_A_MEMORY_SIZE (group 1)またはND_SCARAB_B_MEMORY_SIZE （group2）として、Get_DAQ_Device_Info （NI-DAQ API内）を呼び出します。

メモリのサイズはバイトで返されます。

20 MHzパターンI/O:Digital Clock Config VIまたはDIO Config VIを実行すると、クロック周波数を20MHzに設定できます。C言語でプログラミングをしている場合、timebase = -3およびinterval = 1としてDIG_Block_PG_Configを呼び出してください。

NI 660Xの有効範囲に関する情報

  • DIG_In_Line
  • DIG_In_Prt
  • DIG_Line_Config

  • DIG_Prt_Config

PCIデバイスとWindows 98における割り込み性能の問題

割り込み処理は、以下のすべての条件に適合するシステム上で遅くなることがあります。

  • コンピュータにUSBポートがある。
  • コンピュータがWindows 98のリリース版で動作している。
  • PCIデータ集録デバイスを使用している。
  • データ集録アプリケーションは割り込みを頻繁に発生させる操作を実行している (たとえばハンドシェイクやメッセージ処理など)。
  • PCIデバイスの割り込みレベル（IRQL）がUSBホストコントローラのIRQLと一致する（「PCI DAQデバイスとUSB/1394ホストコントローラのIRQLを決定する」を参照）。

以下のすべての条件に適合すると、極端な遅れ、タイムアウト、またはロックアップがシステム上で起こる可能性があります。

  • コンピュータに1394アダプタポートがある。
  • コンピュータがWindows 98のリリース版で動作している。
  • PCIデータ集録デバイスを使用している。
  • データ集録アプリケーションは、割り込みを頻繁に発生させる操作 (たとえば、アナログソフトウェアトリガ、非同期オカーレンス、その他のメッセージ処理など) を実行している。
  • PCIデバイスの割り込みレベル（IRQL）が1394ホストコントローラのIRQLと一致する（「PCI DAQデバイスとUSB/1394ホストコントローラのIRQLを決定する」を参照）。

PCI DAQデバイスとUSB/1394ホストコントローラのIRQLを決定する

PCI DAQデバイスおよびUSB/1394ホストコントローラのIRQLが同じかどうかを確認するには、以下の手順に従ってください。

  1. 通常画面の左上にあるマイコンピュータアイコンを右クリックします。
  2. ポップアップメニューからプロパティを選択し、システムのプロパティウィンドウを開きます。
  3. デバイスマネージャタブをクリックします。
  4. コンピュータをダブルクリックして「コンピュータのプロパティ」ウィンドウを開きます。
  5. 割り込み要求 (IRQ)が選択されていることを確認します。
  6. 選択ボックスをスクロールし、PCI DAQデバイスを探します。見つかったら、設定の欄にある番号を書き止めます（デバイスのIRQ）。
  7. 選択ボックスをスクロールし、USB/1394ホストコントローラが見つかったら、設定の欄にあるIRQを書き止めます。
  8. PCI DAQデバイスの番号をUSB/1394コントローラの番号と比較します。

PCIデータ集録デバイスの設定番号がUSB/1394ホストコントローラと一致する場合は、以下の方法で、割り込み処理の性能を向上することができます。

メモ：      USB/1394ホストコントローラとPCIデータ集録デバイス間でIRQLを共有すると、割り込み処理の速度にのみ影響します。DMAや他のI/Oに対する影響はありません。また、IRQLをPCIデータ集録デバイスまたは他のPCIデバイス（ビデオやネットワークアダプタなど）間で共有しても、割り込み処理速度には影響しないはずです。

  2. ポップアップメニューからプロパティを選択し、システムのプロパティウィンドウを開きます。
  3. デバイスマネージャタブをクリックします。

USBおよび1394のサポート条件と制約

USBおよび1394に対するオペレーティングシステムのサポート

NI-DAQ 6.7では、USBおよび1394へのソフトウェアサポートはWindows 2000/XP/Me/98搭載のシングルプロセッサシステムのみです。他のバージョンのWindowsを実行している場合、USBや1394デバイスを使用するには、オペレーティングシステムのアップグレードが必要です。

USBと1394に対するナショナルインスツルメンツのソフトウェアのサポート

  • USB DAQデバイスを使用するには、NI-DAQ バージョン6.5以降が必要です。
  • 1394 DAQデバイスを使用するには、NI-DAQ バージョン6.6以降が必要です。
  • USBまたは1394を使用するには、LabVIEW 4.0以降が必要です。

起こりうる問題点

USBまたは1394 DAQPadをPCに接続し、電源を入れてから、DAQアプリケーションやMeasurement & Automation Explorer を起動してください。DAQアプリケーションをロードした後にDAQPadを接続したり電源を入れたりした場合、LabVIEWの中にあるDevice Reset.viを実行するか、あるいはNI-DAQ APIにあるInit_DA_Brds()を呼び出して、DAQPadを使用する前にソフトウェア上でデバイスをリセットしてください。

手動によるリセットに対応していないDAQアプリケーションを使用している場合は（Virtual Benchなど）、アプリケーションを実行する前に、USBまたは1394 DAQPadを接続し電源をオンにする必要があります。

Windowsのエクスプローラ（Internet Explorerを含む）ウィンドウが開いていると、ごくまれにUSBまたは1394デバイスがMeasurement & Automation Explorer に表示されない場合があります。この問題が発生したときは、エクスプローラウィンドウをすべて閉じ、<F5>を押してMeasurement & Automation Explorer のウィンドウをリフレッシュしてください。問題が引き続き起こる場合は、コンピュータを再起動してください。

USBおよび1394 DAQPadは、Windows 2000/XP/Me/98のサスペンドモードをサポートしていません。

構成と電源の管理

USBおよび1394デバイスは、プラグアンドプレイ構成とホットスワップをサポートしています。また、Windows Me/98は電源の管理をサポートします。これらの機能によってデバイス構成は容易になりますが、構成における問題を防ぐには以下のガイドラインに従ってください。

場合によっては、データ集録中に複数の1394デバイスの接続を同時に切ると、問題が発生することがあります。デバイスの接続してから、あるいは接続を切ってから、データ集録ソフトウェアをロードすることが重要です。

デバイスの接続を切った後や電源をオフにした後は、デバイスを再びオンにするまで5〜10秒（または砂時計アイコンが消えるまで）待つ必要があります。

コンピュータによっては、電源の管理がUSBまたは1394デバイスを使用するアクティブなDAQアプリケーションに干渉することがあります。コンピュータの無人運転中にDAQアプリケーションを実行しなければならない場合、スクリーンセーバーだけではなく、オペレーティングシステムやBIOSにおいて電源の管理を無効にしてください。

Windows XP/Me/98で電源の管理を無効にするには、以下の手順に従ってください。

  1. コントロールパネルを開きます（スタート→設定→コントロールパネルをクリックします）。
  2. 電源の管理アイコン（Windows XPでは電源オプション）をダブルクリックし、電源の管理アプレットを起動します。次に電源設定タブをクリックします。
  3. 電源設定リストボックスで、常にオンを選択します。
  4. 「常にオン」の電源設定で、パラメータを以下のように設定します。
    1. システムのスタンバイ：なし（Windows XPでは、休止状態タブを選択し、休止状態を無効にします）
    2. モニタの電源を切る：どれでも選択可能
    3. ハードディスクの電源を切る：なし
  5. OKをクリックして設定を保存し、電源の管理アプレットを閉じます。
  6. コントロールパネルを閉じます。

電源の管理機能がBIOSに組み込まれているかどうか、そしてBIOSの電源管理機能を無効にする手順を確認するには、次の手順に従ってください。

スクリーンセーバーを無効にするには、次の手順に従ってください。

  1. Windowsのデスクトップ上の任意の場所で右クリックし、ポップアップメニューからプロパティを選択してディスプレイのプロパティを開きます。
  2. スクリーンセーバータブをクリックします。
  3. スクリーンセーバータブで、なしを選択します。
  4. OKをクリックして設定を保存して、画面のプロパティウィンドウを終了します。

性能

USB/1394の性能は、CPU速度やメモリアーキテクチャ、システムチップセットの特性など、システム固有の要素によって大きく影響されます。USBおよび1394の操作はOSに対し、ソフト的に負荷が大きいため、データ集録操作中に他のアプリケーションを実行すると、性能に影響が出ることがあります。高速USBおよび1394の操作も、他のアプリケーションの性能に影響を及ぼす可能性があります。

PCに接続されているUSBおよび1394デバイスはすべて、同じ帯域幅を共有しています。複数のUSBおよび1394デバイスを同時に使用すると、DAQアプリケーションの性能に影響が出ることがあります。また、PCのUSBおよび1394ホストコントローラはPCIバスに常駐しているため、他のPCIバス帯域幅を多く使うPCIデバイスを使用すると性能に影響が出る可能性があります。

データはUSBおよび1394バス上をパケット単位で転送されるため、少量または正確な量のデータを転送する操作のタイミングは、プラグインDAQデバイスの場合ほど決定的ではありません。たとえば、メッセージ処理や割り込みを頻繁に使用する操作のタイミングは、USBおよび1394デバイスの場合より大きく変化することが考えられます。

SCXI_Calibrateの注意事項

SCXI_Calibrateを使用して内部キャリブレーションを実行する前に、スキャン操作を行った場合は、-10636エラーが発生する可能性があります（同期関数でダブルバッファ操作を開始できないため）。C言語で開発している場合、キャリブレーションを行うDAQデバイスの番号とDbMode = 0をDAQ_DB_Config関数へ渡して、関数を呼び出します。LabVIEWを使用している場合、DAQデバイスのDevice Reset VIを実行してください。また、一度、開発環境を終了して、再起動することもできます。

リモートSCXIデバイスの使用に関する情報

SCXI-1104またはSCXI-1125をリモートSCXI構成で使用する場合は、SCXI-2000シャーシまたはSCXI-2400モジュールのファームウェアを更新する必要があります。リモートSCXIを持っていてSCXI-1104またはSCXI-1125を使用する予定がない場合は、更新の必要はありません。

ファームウェアの更新を行うには、次の手順に従ってください。

  1. リモートSCXIが正しく機能していてオンになっていることを確認します。
  2. Measurement & Automation Explorer のアイコンをダブルクリックし、Measurement & Automation Explorer を開きます。
  3. デバイスとインタフェースをクリックして、デバイスツリーを表示します。
  4. プロパティを選択します。SCXI-2000シャーシを使用している場合は、そのフォルダを右クリックしてプロパティを選択します。SCXI-2400モジュールを使用している場合、そのモジュールが入っているシャーシのフォルダをダブルクリックしてから、SCXI-200フォルダを右クリックしてプロパティを選択します。
  5. 再プログラムをクリックします。
  6. 更新されたリモートSCXIイメージファイルへのパスを入力し、OKをクリックします。

新しいファームウェアがご使用のリモートSCXIシャーシまたはモジュールにダウンロードされます。ケーブルをはずしたり、この操作を中断したりしないでください。

リモートデバイスアクセス (RDA)の使用に関する情報

RDAアプリケーションでNI-DAQ 6.9を使用するには、クライアント（ローカル）とRDAサーバ（リモート）のコンピュータ両方にNI-DAQ　6.9を必ず使用してください。これは、このバージョンに追加された新しいRDA保護機能のために必要です。

この保護機能により、サーバで構成されている各デバイスのパスワードを指定することができます。これは、そのデバイスへのRDAを正しいパスワードを知っているクライアントに限定するためです。デバイスのパスワードを設定するには、以下の手順に従います。

  1. サーバのコンピュータで、Measurement & Automation Explorer を開きます（デスクトップのMeasurement & Automation Explorer のアイコンをダブルクリックします）。
  2. デバイスとインタフェースをダブルクリックして、デバイスツリーを展開します。
  3. アクセスを制限したいデバイスを右クリックし、ポップアップメニューからプロパティを選択して、「デバイスの構成」ウィンドウを開きます。
  4. リモートアクセスタブをクリックします。
  5. パスワードの設定をクリックして「パスワードの設定」ダイアログボックスを開き、パスワードを入力します。

デバイスのパスワードを設定した後、RDAを使用してデバイスを使用するには、RDAのクライアントは正しいパスワードを指定する必要があります。デフォルトでは、パスワードは空白です。これは、デバイスはパスワードにより保護されていないことを示します。パスワードの変更時に、クライアントがRDAデバイスを使用している場合、正しいパスワードが設定されるまで-10340エラーが発生します。デバイスの正しいパスワードを設定するには、以下の手順に従います。

  1. クライアントのコンピュータで、Measurement & Automation Explorer を開きます（デスクトップのMeasurement & Automation Explorer のアイコンをダブルクリックします）。
  2. デバイスとインタフェースをダブルクリックして、デバイスツリーを展開します。
  3. アクセスを制限したいデバイスを右クリックし、ポップアップメニューからプロパティを選択して、「デバイスの構成」ウィンドウを開きます。
  4. デバイスの正しいパスワードを入力します。

サーバに設定できる各デバイスレベルのパスワードに加えて、ルートレベルのパスワードがあります。これを設定すると、アクセス権限のないユーザがRDAサーバをリモート構成するのを防ぐことができます。サーバのコンピュータでこのパスワードを設定するには、サーバでMeasurement & Automation Explorer を開き、ツール→NI-DAQ構成→リモート構成パスワードの設定を選択します。このパスワードを設定した後、Measurement & Automation Explorer を通じてサーバをリモート構成しようとする（ツール→NI-DAQ構成→リモートDAQ構成をクリックして）クライアントは、まず、正しいパスワードを入力する必要があります。ルートレベルの正しいパスワードを知っているクライアントは、デバイスレベルの各パスワードと同様、リモートDAQ構成でそのサーバのルートレベルのパスワードを変更することができます。

NI-DAQ6.7以降では、RDAにはタイムアウトの機能が備わっています。RDAクライアントコンピュータがRDAサーバコンピュータからの応答を待機している間、ネットワーク接続を失ったときに、この機能が役立ちます。タイムアウト機能がなければ、RDAクライアントは永久に待機し、LabVIEWのプログラムがハングアップしたように見えます。タイムアウト機能があれば、LabVIEWは-10340（noConnectError）を返します。

タイムアウトのデフォルト値は240,000秒（4分）です。これを変更するには、既存のレジストリキーに値を追加する必要があります。以下の手順で、タイムアウト値を変更します：-{}-

  1. Regedit.exeまたはregedt32.exeを実行します。
  2. HKEY_LOCAL_MACHINE→SOFTWARE→National Instruments→NI-DAQ→Current Version→Configuration Managerを開きます。
  3. 編集メニューから新規→DWORD値を選択します。
  4. 新しい値の名称をReceiveTimeoutValueにします。
  5. ReceiveTimeoutValueをダブルクリックして、データ値を入力するダイアログボックスを開きます。ミリ秒単位でタイムアウトを入力します。基数が10進であることを確認します。タイムアウトを0にすると、NI-DAQ6.7以前のバージョンのように無限に待機するように設定されます。
  6. Regedit.exeまたはregedt32.exeを終了します。

データの読み取りまたは書き込みをするNI-DAQの操作は指定された速度とデータの量により、かなりの時間がかかる場合があることに注意してください。10スキャン/秒でデータを集録中に、AI Buffer Read VIを呼び出し、100スキャン読み取るには、10秒かかります。ReceiveTimeoutValueが10000ミリ秒未満に設定されていると、ネットワークの接続が正常でも、-10340エラーが発生します。

Measurement & Automation Explorer でDAQ構成ファイルを修正する際の注意点

Measurement & Automation Explorer のバージョン2.0から2.2では、ツール→NI-DAQ構成の中にあるメニューオプションを使用して、DAQ構成ファイルを操作します。アクティブ構成を設定オプションで、DAQ構成ファイルをアクティブファイルに設定します。別名で構成を保存オプションは、現在の構成ファイルを新しいファイルとして保存します。新規構成を作成オプションは、新しい構成ファイルを、ファイル名を選択して作成します。

Visual Basicで16-Bitと32-BitのUnsigned Integerパラメータを渡す

Unsigned 16-bitパラメータ：Visual BasicのINTEGERタイプはsigned値で、32,767よりも大きい値は、負の値として考えられます。解決方法は、下記のとおりです：

  1. 関数がunsigned 16-bit integerを返す場合、負の数を読み戻し、その負の数に5,336を追加します。

例えば：

Dim myUnsignedCnt (Long)

Dim RetCnt (Integer)

`（retCntは呼び出された関数から、返される値です）

`unsigned 16-bit値を返すDAQ関数を呼び出して、retCntに格納することができます。

if retCnt < 0 then

myUnsignedCnt = CInt(retCnt) + 65536

else

myUnsignedCnt = retCnt

end if

  1. unsigned 16-bit値をunsigned 16-bitタイプのパラメータと一緒の関数に渡す場合、関数に渡す値を最初に計算する必要があります。

例えば：

Dim myUnsignedCnt (Long)

Dim RetCnt (Integer)

`関数に渡す値を計算して、myUnsignedCntに格納します。

if myUnsignedCnt > 32767 then

ToPassVal = CInt(myUnsignedCnt - 65536)

else

ToPassVal = CInt(myUnsignedCnt)

end if

Unsigned 32-bitパラメータ：Visual BasicのLONG integerタイプはsigned32-bitタイプで、2,147,483,647よりも大きい値は、負の値として考えられます。解決方法は、下記のとおりです：

  1. 関数がunsigned 32-bit integerを返す場合、負の数を読み戻し、その負の数に4,294,967,296を追加します。

例えば：

Dim myUnsignedCnt (Double)

Dim retCnt (Long)

`（retCntは呼び出された関数から、返される値です）

`unsigned 16-bit値を返すDAQ関数を呼び出して、retCntに格納することができます。

if retCnt < 0 then

myUnsignedCnt = CInt(retCnt) + 4294967296

else

myUnsignedCnt = retCnt

end if

  1. unsigned 32-bit値をunsigned 32-bitタイプのパラメータと一緒の関数に渡す場合、関数に渡す値を最初に計算する必要があります。

例えば：

Dim myUnsignedCnt (Double)

Dim ToPassVal (Long)

`関数に渡す値を計算して、myUnsignedCntに格納します。

if myUnsignedCnt > 2147483647 then

ToPassVal = CLng(myUnsignedCnt - 4294967296)

else

ToPassVal = CLng(myUnsignedCnt)

end if
